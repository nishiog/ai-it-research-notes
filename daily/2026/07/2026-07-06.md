## 1. Claude Opus 4.8 / Sonnet 5 で "tool calling" の信頼性が低下 — Armin Ronacherの分析

**一次情報:** [Better Models: Worse Tools — lucumr.pocoo.org](https://lucumr.pocoo.org/2026/7/4/better-models-worse-tools/)(2026-07-04)

エージェント型コーディングツールPiの開発者Armin Ronacher氏が、Claude Opus 4.8 / Sonnet 5がツール呼び出し時にスキーマに存在しない架空のフィールド(`requireUnique`、`oldText2`など)を`edits[]`配列に混入させ、スキーマ違反で呼び出しが拒否される不具合を報告した。実際の編集内容(`oldText`/`newText`)自体は正確なのに、末尾の余計なキーのせいで壊れるという。最新モデルほど悪化するという逆説的な現象で、古いモデルでは発生しないとのこと。

氏の推測では、Claude Code(またはそれに酷似したハーネス)を使った訓練が原因。Claude Code自身のクライアントは不正なツール呼び出しに寛容な修復処理(エイリアス許容、型強制、未知キーのフィルタリング等)を行うため、強化学習の過程で多少不正確な呼び出しでもタスクが完了・報酬が得られてしまい、誤りを罰する勾配がほとんど働かない。結果としてモデルはClaude Code特有のフラットな編集ツール形状に強い先入観を持ち、ネスト構造を持つ他社ツールのスキーマに対して毎回でたらめなフィールドを生成してしまう、と分析している。対策としてAnthropicの`strict`モード(スキーマ外キーのサンプリングをサーバー側で禁止)を有効にすると解消したと報告されているが、複雑なツール定義では`strict`モードがAPIエラーを起こす制約があり、Claude Code自身はこれを採用していないと氏は推測している。

**Xでのシグナル:** [@TirunagariRohan の投稿](https://x.com/TirunagariRohan/status/2073739298831220873)は「20% failure rate on a simple edit tool」と紹介していたが、具体的な失敗率の数値はブログ本文には明記されておらず、投稿側の要約表現である点に留意。

## 3. Claude Code の Skills / Plugin marketplace エコシステムが拡大

**一次情報:** [anthropics/skills — GitHub](https://github.com/anthropics/skills)

Anthropic公式リポジトリ`anthropics/skills`(約158kスター)が、Claude向け「Agent Skills」の参照実装として公開されている。各スキルは`SKILL.md`(YAMLフロントマターに`name`/`description`のみ必須)を含むフォルダとして定義され、Word/Excel/PowerPoint/PDF操作を行う`document-skills`はプロダクションで使われている実装がソースアベイラブルとして公開されている。導入は次の2コマンドで完了する。

```
/plugin marketplace add anthropics/skills
/plugin install document-skills@anthropic-agent-skills
```

この公式リリースと前後して、企業・コミュニティ製のskillsマーケットプレイスも相次いで登場している(詳細はTips参照)。「汎用マーケットプレイス」よりも「特定の垂直領域(決済、.NET、LMS、画像生成ワークフロー等)に特化したskills/MCPサーバー」の方向にエコシステムが広がっている点が実務上のポイント。

## 4. Model Context Protocol TypeScript SDK 2.0.0-beta.2 — CommonJS対応

**一次情報:** [modelcontextprotocol/typescript-sdk — Releases](https://github.com/modelcontextprotocol/typescript-sdk/releases)(GitHub, 2026-07-02)

`server`, `client`, `core`, `node`, `hono`, `fastify`, `express`, `codemod`等の主要パッケージが`2.0.0-beta.2`として一斉リリースされた。目玉はPR #2405によるCommonJSビルドの追加で、`exports`マップに`require`条件が加わり、既存のCommonJSベースNode.jsアプリからも`require('@modelcontextprotocol/...')`で読み込めるようになった(ESM/CommonJS両対応、拡張子は`.mjs`/`.cjs`)。そのほか、`MissingRequiredClientCapabilityError`発生時に一部ケースでHTTP 200を誤って返していた不具合がHTTP 400に修正され(PR #2399)、v1→v2移行用の`codemod`も実際の移行作業で得た知見を反映して改善されている(PR #2412)。

**Xでのシグナル:** [@Bell_QuoLu の投稿](https://x.com/Bell_QuoLu/status/2073678512519786649)、[@hummusonrails の投稿](https://x.com/hummusonrails/status/2073783515225940424)(MCPクライアント/サーバー保守者に検証観点を問う投稿)。
